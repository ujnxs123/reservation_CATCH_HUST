# HUST 场馆预约客户端

## 核心目标

- 在预约开放前完成账号登录、Cookie 保存和会话验证。
- 使用 NTP 与服务器 Date 头修正本地时间误差。
- 在目标时间前保持 CAS / 预约系统会话可用。
- 从预约页解析 token、csrf、场地片段和可提交 payload。
- 通过 `requests.Session` 执行轻量级锁单请求，减少浏览器渲染开销。
- 锁单成功后切回浏览器完成支付页验证码和最终提交。
- 对失败、超时、会话过期、重登录和多场馆切换做明确处理。

## 工作流程

```text
读取配置
  -> 选择账号与浏览器 profile
  -> 登录 CAS 并保存 storage_state
  -> 检查电子账户余额
  -> 等待登录刷新时间
  -> 静默保活 / 刷新登录态
  -> 等待预约开放时间
  -> 解析场馆剩余页面
  -> 获取实时空闲状态和有效 token
  -> 锁单
  -> 浏览器进入支付页
  -> 点选验证码识别与提交
  -> 返回预约结果
```

## 核心设计

### 浏览器负责稳定上下文

登录、CAS 跳转、2FA、Cookie 保存和预约页可达性验证由 Playwright 处理。这样可以尽量贴近真实浏览器环境，减少因为缺少上下文导致的异常跳转。

### HTTP 负责关键速度

真正进入锁单阶段后，流程会从 `storage_state` 克隆出 `requests.Session`，复用浏览器保存的 Cookie，并用 HTTP 请求完成。

### token 按阶段刷新

`syqk` 页面中的 token 只用于本轮 `getsyzt`，而 `getsyzt` 返回的新 token 才是 `step2` 需要的有效 token。代码会在每轮请求后更新 payload，避免复用过期 token。

### 单 pian 顺序锁单

当前实现会从实时空闲结果中取第一个可用 `pian`，并以顺序方式锁单。这样做牺牲了一点激进并发，但降低了同一 Cookie 下重复撞单和状态混乱的概率。

### 会话过期可恢复

如果 HTML 解析不到 token / pian，或 `getsyzt` 返回的不是预期 JSON，流程会把它识别为可能的 session 过期，并尝试重新登录后继续当前场馆。

### 验证码放在锁单之后处理

锁单成功会返回支付页地址。客户端再打开浏览器进入支付页，捕获 `captcha/get` 响应，识别点选验证码，提交校验并最终提交订单。

## 配置

本地配置文件是 `config.jsonc`。所有本地配置不会上传远程服务器，均在本地保存。

示例结构：

```jsonc
{
  "startTime": "18:00:00",
  "endTime": "20:00:00",
  "partnerType": 1,
  "payType": "-1",
  "TARGET_VENUE_IDS": [69, 45],
  "accounts": [
    {
      "name": "default",
      "student_id": "你的学号",
      "password": "你的密码",
      "profile_dir": "./profiles/default"
    }
  ],
  "max_retries": 2,
  "retry_sleep_min": 0.8,
  "retry_sleep_max": 1.8,
  "max_relogin_during_booking": 2

## 当前限制

- 不能保证百分百预约成功，结果仍受账号状态、网络、服务器响应、场地余量和平台规则影响。
- 当前锁单策略偏保守，默认单 pian 顺序执行，不做激进并发。
- 会话文件、账号配置和日志都可能包含敏感信息，应仅保存在可信本机环境。
